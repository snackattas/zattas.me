# Skills

<h2 className="not-prose mt-8 !text-xl sm:!text-xl font-normal tracking-tight text-zinc-900 dark:text-zinc-50">
  Focus areas
</h2>

<div className="mt-6 not-prose">
  <PillGrid>
    <DetailPill title="Test reporting">
      <ul>
        <li>Make reports easy to scan: clear sections, minimal noise.</li>
        <li>Error output should include next steps + what system to investigate.</li>
        <li>Log key test context (e.g. account identifiers) for fast debugging.</li>
        <li>Persist run artifacts (e.g. generated files) for post-mortems.</li>
      </ul>
    </DetailPill>

    <DetailPill title="Test data strategy">
      <ul>
        <li>"Sign" factory data with metadata so it can be identified and cleaned up later.</li>
        <li>Enable environment-wide cleanup without paying the cost per test.</li>
        <li>Allow product code to behave differently for signed test data when appropriate.</li>
      </ul>
    </DetailPill>

    <DetailPill title="Frontend test ergonomics">
      <ul>
        <li>Use browser cookies/local config to toggle safe UI mocks when needed.</li>
        <li>Mock expensive or flaky third-party UI integrations in lower envs.</li>
        <li>Make UI timing behavior test-friendly (e.g. non-expiring toasts).</li>
      </ul>
    </DetailPill>

    <DetailPill title="Waiting for async work">
      <ul>
        <li>Prefer deterministic waits: DB checks, API polls, or explicit status endpoints.</li>
        <li>Reload/retry patterns where appropriate (but keep timeouts bounded).</li>
        <li>Sometimes the right move is adding a purpose-built testing API.</li>
      </ul>
    </DetailPill>

    <DetailPill title="Service health checks">
      <ul>
        <li>Before running tests, verify dependent services via their <code>/status</code> endpoints.</li>
        <li>Fail early with a clear error so the root cause is obvious.</li>
      </ul>
    </DetailPill>

    <DetailPill title="Auth in E2E tests">
      <ul>
        <li>Log in via API and inject tokens into the browser session.</li>
        <li>Keep a single end-to-end UI login test to validate the real flow.</li>
      </ul>
    </DetailPill>
  </PillGrid>
</div>
